Conjoint Secrets: What Customers Really Value

Customers don't evaluate price in isolation — they trade off price against features, brand, speed, and a dozen other attributes simultaneously. Choice-Based Conjoint (CBC) analysis decomposes these trade-offs statistically, revealing the part-worth utility of every attribute level, the willingness-to-pay for each feature, and the optimal product configuration for maximum market share.

Part 1: The Feature Pricing Problem

A product team debates whether to add AI-powered analytics to their SaaS platform. Engineering cost: $2M. The PM thinks customers will pay $30/month more. Sales thinks $10 at best. Marketing wants to include it free as a differentiator. Nobody has data. This is exactly the problem conjoint analysis solves: how much is each feature worth to the customer, measured in dollars?

What Conjoint Does That Direct Questions Can't

Ask customers "would you pay $30 more for AI analytics?" and they'll say no (strategic bias: they want it cheap). Ask "what's the most you'd pay?" and they'll lowball. But show them a series of product configurations with different feature/price combinations and ask "which would you choose?", and their revealed preferences expose their true trade-offs. Conjoint extracts willingness-to-pay without ever directly asking "how much would you pay?"

Part 2: Choice-Based Conjoint (CBC)

In CBC, respondents repeatedly choose between 2-4 product alternatives (plus a "none" option). Each alternative is a bundle of attribute levels. By varying which levels appear together, we can statistically decompose the total utility into part-worth utilities per attribute level.

RANDOM UTILITY MODEL (McFadden, 1974)
Uij = Vij + εij = ∑k βk · xijk + εij
U = total utility | V = systematic (estimated) | ε = random | βk = part-worth for attribute level k

The probability that respondent i chooses alternative j from choice set C follows the multinomial logit:

MULTINOMIAL LOGIT CHOICE PROBABILITY
P(choice = j) = exp(Vj) / ∑m ∈ C exp(Vm)

From Part-Worths to Willingness-to-Pay

WILLINGNESS TO PAY FOR A FEATURE
WTPfeature = − (βfeature − βbase) / βprice
βprice = utility per dollar (must be negative) | This converts utility units to dollar units
The $18/month Answer

Using the formula above, if the part-worth for "AI analytics = yes" is 0.45 and "AI analytics = no" is 0 (base), and the price coefficient is -0.025 per dollar, then WTP = -(0.45 - 0) / (-0.025) = $18/month. The PM was too optimistic, Sales was too conservative, and Marketing's "give it away free" idea would destroy $18/month of value per customer.

Part 3: Experimental Design

Number of attributes4-7More than 7 creates cognitive overload; respondents start ignoring attributes
Levels per attribute2-5More levels = more precision but requires larger sample
Alternatives per choice set3-4 + "none"Fewer is easier for respondents; more is more informative
Choice tasks per respondent8-15Below 8 = insufficient data; above 15 = fatigue
Sample size200-500Rule of thumb: 500/(# alternatives x # tasks) per level
Design typeD-optimal or balanced overlapMaximizes information per choice task
Price as an Attribute

Always include price as an attribute in conjoint. Use 3-5 price levels spanning a realistic range (typically ±30% of current price). The coefficient on price is what converts part-worths to WTP. Without price, you can rank features by importance but can't monetize them.

Part 4: Market Simulation

The real power of conjoint isn't the part-worths — it's market simulation. Once you have the estimated model, you can simulate any product configuration against any competitive set and predict market shares.

Simulation QuestionHow to Answer
"What market share do we get at $99 vs. $119?"Define your product at each price, define competitors, compute logit shares
"What if we add AI analytics at +$20?"Add the feature's part-worth, increase price, recompute shares
"What's the optimal configuration?"Enumerate feasible combinations, compute share x margin for each
"What if a competitor launches a new feature?"Add competitor product to choice set, see how shares shift

Part 5: Implementation in Python

Python · Choice-Based Conjoint
import pandas as pd
import numpy as np
from sklearn.linear_model import LogisticRegression
import pylogit  # pip install pylogit

choices = pd.read_csv('conjoint_choices.csv')
# Long format: respondent_id, task_id, alt_id, chosen,
# price, brand_A, brand_B, feature_ai, feature_support, speed_fast

# === Multinomial Logit via pylogit ===
spec = OrderedDict()
spec['intercept'] = [1, 2]  # ASCs for alts 1, 2 (3 = base)
names = OrderedDict()
names['intercept'] = ['ASC 1', 'ASC 2']

for var in ['price', 'brand_A', 'feature_ai',
            'feature_support', 'speed_fast']:
    spec[var] = [[1, 2, 3]]
    names[var] = [var]

model = pylogit.create_choice_model(
    data=choices, alt_id_col='alt_id',
    obs_id_col='task_id', choice_col='chosen',
    specification=spec, names=names,
    model_type='MNL')
model.fit_mle(np.zeros(len(spec)))
model.get_statsmodels_summary()

# === Willingness to Pay ===
b_price = model.params['price']
for feat in ['feature_ai', 'feature_support', 'speed_fast']:
    wtp = -model.params[feat] / b_price
    print(f"WTP for {feat}: ${wtp:.2f}/month")

# === Market Simulation ===
def simulate_shares(products, params):
    """products: list of dicts with attribute levels"""
    utilities = []
    for p in products:
        v = sum(params.get(k, 0) * v for k, v in p.items())
        utilities.append(v)
    exp_v = np.exp(utilities)
    return exp_v / exp_v.sum()

our_product = {'price': 99, 'feature_ai': 1, 'speed_fast': 1}
competitor  = {'price': 89, 'feature_ai': 0, 'speed_fast': 1}
shares = simulate_shares([our_product, competitor], model.params)
print(f"Predicted shares: Us={shares[0]:.1%}, Them={shares[1]:.1%}")
