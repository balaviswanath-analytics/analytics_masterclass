Core Principle

The Containment Imperative

Guiding Principle

Contain first, analyze second. Every day of customer-impacting exposure while the team discusses root causes is a day of avoidable damage. Containment is not a workaround — it is a professional obligation.

Containment actions are temporary measures designed to stop or reduce customer impact while the permanent root cause analysis proceeds. They are not solutions — they are bought time. A containment that masks evidence or prevents proper diagnosis is worse than no containment at all.

What Makes a Good Containment Action

CharacteristicDescription & Why It Matters
ReversibleCan be undone quickly once the permanent fix is deployed. Containments that become permanent create technical debt and hide systemic problems.
MonitoredImpact and effectiveness are tracked continuously. If containment is not working, you need to know immediately — not at the next weekly review.
Non-destructiveDoes not alter or destroy evidence needed for root cause analysis. Avoid containment actions that flush logs, overwrite states, or clear queues without backups.
DocumentedClear record of what was done, when, by whom, and what it was designed to achieve. Undocumented containment creates confusion in D6 when it's time to remove it.
Time-boundHas a planned removal date or condition tied to permanent fix validation. Open-ended containment is a containment that will never be removed.
Containment Strategies

Common Containment Approaches by Type

StrategyWhen to UseExamples
Traffic Rerouting System or service degradation affecting specific nodes/regions Route traffic to backup data center • Shift load to healthy nodes • Use alternate API endpoint
Feature Toggles New feature causing issues — need to disable without full rollback Disable problematic feature flag • Rollback to previous version • Enable only for test users
Manual Overrides Automation failure requiring human intervention temporarily Manual approval workflows • Operator-assisted processing • Bypass automated checks temporarily
Rate Limiting Capacity or performance issues from volume spikes Throttle API calls • Queue non-critical requests • Implement request prioritization
Data Corrections Data quality issues causing downstream failures Run correction scripts • Manual data cleanup • Block corrupt data ingestion
Domain Examples

Containment in Action: By Industry

Telecom

eSIM Activation Failures • Circle X

Route eSIM activations to legacy EUM provider for Circle X customers until root cause is fixed. Monitor activation success rate hourly. Remove routing rule once permanent fix is validated and stable for 48 hours.

E-Commerce

Checkout Failures • Safari Mobile

Display alternative payment flow for Safari users. Add browser detection to route Safari traffic to legacy payment gateway. Track conversion rates separately for contained vs normal flow to measure effectiveness.

Banking

Auto-Debit Failures • Post-Migration

Manually process failed auto-debits in overnight batch. Notify affected customers proactively of manual processing. Run daily reconciliation report until permanent fix is deployed and verified.

Monitoring

Tracking Containment Effectiveness

Every containment action must have defined metrics being tracked in near real-time. If containment is not measurably working within the first few hours, escalate — do not wait for the daily standup.

Primary KPI
Return to 90%+ of baseline or agreed threshold
Side Effects
Latency, throughput, or error rate changes from containment
Customer Impact
Support ticket / escalation volume reduction
Business Impact
Revenue recovery or loss reduction rate
Ops Cost
Manual effort hours per day for maintained containment
Critical Rule

Containment must not alter or destroy evidence needed for D4 root cause analysis. Before deploying containment, explicitly confirm: what logs, states, or data must be preserved? Designate an analyst to capture pre-containment baselines.

Exit Criteria for D3

All of the following must be true before D4 root cause analysis is considered fully launched:

  • Containment action implemented, active, and confirmed working
  • Monitoring metrics defined and actively tracked (near real-time)
  • Removal plan documented with specific conditions or timeline
  • Stakeholders notified of temporary containment measure and what to expect
  • Evidence preserved: logs, snapshots, states captured before containment altered behavior
