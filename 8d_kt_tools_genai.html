SQL

SQL for Evidence Gathering

SQL is the primary tool for extracting evidence to populate KT analysis tables. The two most useful queries in the 8D workflow are IS/IS NOT segmentation and Change Analysis lookup.

IS / IS NOT Population Query — Use in D4 to find affected vs unaffected segments
SELECT
    customer_segment,
    os_version,
    circle_name,
    COUNT(*) AS total_attempts,
    SUM(CASE WHEN status = 'failed' THEN 1 ELSE 0 END) AS failures,
    ROUND(
        100.0 * SUM(CASE WHEN status = 'failed' THEN 1 ELSE 0 END) / COUNT(*),
        2
    ) AS failure_rate
FROM activation_logs
WHERE attempt_timestamp BETWEEN '2026-01-15' AND '2026-01-20'
GROUP BY customer_segment, os_version, circle_name
ORDER BY failure_rate DESC;

-- High failure rates (>20%) → IS column
-- Low failure rates (<5%) → IS NOT column
-- What is DISTINCTIVE about high-failure segments?
Change Analysis Lookup — Use in D4 to surface changes before problem start
SELECT
    change_date,
    change_type,
    component,
    change_description,
    author
FROM change_log
WHERE
    change_date BETWEEN '2026-01-01' AND '2026-01-15'
    AND (
        component LIKE '%activation%'
        OR component LIKE '%eSIM%'
        OR component LIKE '%provisioning%'
    )
ORDER BY change_date DESC;
Python

Python for Hypothesis Testing

Use Python at D4 to quantify correlations and detect structural breaks in time series data. Both examples below produce Tier 2 evidence — pair with Tier 1 (log traces, code diffs) for full validation.

Correlation Analysis — Quantify relationship between EUM timeout and failure rate
import pandas as pd
import numpy as np
from scipy.stats import pearsonr

# Load activation logs and EUM timeout data
df = pd.read_sql(query, connection)

# Calculate correlation between EUM timeout and activation failure
correlation, p_value = pearsonr(
    df['eum_timeout_ms'],
    df['activation_failed']
)

print(f"Correlation: {correlation:.3f}")
print(f"P-value: {p_value:.4f}")

# Interpretation:
# correlation > 0.7 + p_value < 0.05 → strong Tier 2 evidence
# Now pair with log traces (Tier 1) to confirm mechanism
Change Detection — Identify structural break at problem start date in time series
import matplotlib.pyplot as plt
from statsmodels.tsa.seasonal import seasonal_decompose

# Decompose activation success rate time series
ts = df.set_index('date')['activation_success_rate']
decomposition = seasonal_decompose(ts, model='additive', period=7)

# Plot trend, seasonality, and residuals
decomposition.plot()
plt.tight_layout()
plt.show()

# Look for sharp changes in the TREND component at Jan 15
# A step-change in trend is strong evidence of an external cause
Excel

Excel for KT Matrices

Excel remains the most practical tool for building and sharing KT analysis matrices within a team. Structure your workbook as follows:

TAB 1
IS / IS NOT Matrix

4 dimensions with conditional formatting: IS cells green, IS NOT cells red

TAB 2
Change Timeline

Dates, change types, relevance scores with status column (suspect/ruleout)

TAB 3
MPC Scoring Matrix

Hypotheses scored 1–5 on 5 criteria with auto-calculated totals and rank

TAB 4
Prevention Tracker

D7 actions with owner, date, success measure, and completion status

Team Tip

Create a standardized Excel template with pre-formatted tabs for consistent analysis across the team. A team that uses the same template produces work that reviewers can gate-check in half the time.

GenAI Integration

GenAI Prompts for Analysis Acceleration

Critical Rule

GenAI is a thinking partner, not a replacement for evidence-based analysis. ALWAYS validate GenAI outputs with real data. A GenAI-generated hypothesis that is not tested against data is not a root cause — it is a guess with better formatting.

1

Hypothesis Generation from IS/IS NOT

USE WHEN: You have completed IS/IS NOT and need candidate hypotheses to score in MPC
I am conducting a root cause analysis using Kepner-Tregoe methodology. Here is my IS/IS NOT analysis: IS (what IS affected): [paste your IS observations across all 4 dimensions] IS NOT (what is NOT affected): [paste your IS NOT observations] Based on the distinctions between IS and IS NOT, generate 5 potential hypotheses for what might be causing this problem. For each hypothesis: 1. Explain which IS observations it explains 2. Explain which IS NOT observations it does NOT explain (and why) 3. Suggest one test that would confirm or rule out this hypothesis

Expected output: 5 structured hypotheses ready for MPC scoring. Add them to your MPC matrix and score independently before reviewing GenAI output.

2

Change Analysis Support

USE WHEN: You have a change timeline but need help assessing causal relevance
I have a problem that started on [DATE AT TIME]. The problem affects [brief IS description]. Here are the changes that occurred in the 14 days before the problem: [paste your change timeline: date | change type | description] Help me assess which changes are most likely causally related. For each: 1. Rate likelihood (High/Medium/Low) with reasoning 2. Explain the technical mechanism by which it could cause this problem 3. Identify what evidence would confirm or rule it out Rank changes by most probable cause order.

Expected output: Ranked change list with causal reasoning. Cross-reference with IS/IS NOT — the highest-ranked GenAI change should also explain IS/IS NOT distinctions.

3

Root Cause Validation Check

USE WHEN: Before submitting D4 for Reviewer approval — run as a self-check
I believe I have identified the root cause of a problem. Root cause: [YOUR ROOT CAUSE STATEMENT] IS/IS NOT analysis: [paste full IS/IS NOT table] Evidence gathered: [list your evidence items and their tier] Please review: 1. Does this root cause explain ALL IS observations? List any gaps. 2. Does this root cause correctly NOT explain IS NOT observations? 3. Are there alternative explanations I should rule out? 4. Is the evidence sufficient (Tier 1 present, at least 2 sources)? 5. What is the weakest point in my logic?

Expected output: Critical assessment of your root cause logic. The "weakest point" question is most valuable — address it before the Reviewer finds it.

4

Prevention Actions Brainstorming

USE WHEN: Building D7 prevention plan after root cause is validated
Root cause of OCCURRENCE: [paste validated root cause of occurrence] Root cause of ESCAPE: [paste validated root cause of escape] Generate comprehensive prevention actions across these 5 categories: 1. Monitoring enhancements 2. Process changes 3. Automation opportunities 4. Standards / guidelines 5. Training needs For each action: - Specify exactly WHAT needs to change - Identify WHO should own it (by role) - Suggest a realistic implementation timeline - Define a measurable success criterion

Expected output: Draft D7 prevention action list. Filter for feasibility and assign concrete owners before adding to your Prevention Tracker.
