Foundational Principle

Two Root Causes Required

Non-Violable Rule

No root cause, no solution. You must identify and validate BOTH the root cause of occurrence (why it happened) AND the root cause of escape (why detection failed). A solution that addresses only one is incomplete.

Root Cause of OCCURRENCE
  • ?Why did this problem happen?
  • ?What specific condition or change caused the defect?
  • ?What is the technical mechanism?
Root Cause of ESCAPE
  • ?Why did this problem reach customers?
  • ?What detection or prevention mechanism failed?
  • ?Why did testing or monitoring not catch this?
KT Technique 1

IS / IS NOT Analysis

IS/IS NOT forces the team to define the precise boundaries of the problem. You cannot hypothesize a cause until you know exactly what the problem affects and — critically — what it does NOT affect. The contrast is where the clues live.

The Four Dimensions

Complete all four dimensions. Partial IS/IS NOT analysis is a common anti-pattern and a reviewer rejection trigger.

Dimension IS (What IS affected) IS NOT (What is NOT affected)
WHAT eSIM activation failure — profile download step
Which products, features, functions?		 Physical SIM activations are unaffected
What similar items are NOT affected?
WHERE Circle X (Mumbai metro) — all towers in region
Which locations, regions, systems?		 Circles A, B, C are completely unaffected
What locations are spared?
WHEN Since Jan 15, 6pm. Worse during peak hours (9am–6pm)
When did it start? Is there a pattern?		 Before Jan 15. Off-peak hours have lower (but not zero) failure rate
When was it normal?
EXTENT 29% failure rate (15,000 daily attempts → 4,350 failures)
How big is the problem? How many affected?		 71% of activations still succeed
What is NOT being impacted within affected scope?
Key Analytical Insight

After completing IS/IS NOT, ask: What is DISTINCTIVE about the IS cases that is absent in IS NOT cases? This single question drives hypothesis generation. In the example above: only Circle X was upgraded with new RAN nodes on Jan 10. Only eSIM uses the EUM provider that was migrated on Jan 13. These distinctions point directly to hypotheses.

KT Technique 2

Change Analysis

Something changed. If you cannot find the change, you are not looking hard enough. Map every change that occurred in the window before the problem started against the IS/IS NOT dimensions.

Types of Changes to Investigate

Systematically audit all six change categories. Analysts often miss Process and Environment changes while focusing only on Code changes.

CODE / CONFIG

Deployments, config changes, feature flags, API updates

INFRASTRUCTURE

Server upgrades, network changes, capacity, routing updates

DATA

Schema changes, migrations, reference data updates, volume spikes

PROCESS

Workflow changes, policy updates, team restructuring

ENVIRONMENT

External system changes, vendor updates, third-party integrations

LOAD / DEMAND

Traffic patterns, user behavior, seasonal factors, campaigns

KT Technique 3

Most Probable Cause (MPC) Scoring

After IS/IS NOT and Change Analysis, you will have multiple candidate causes. MPC scoring uses weighted criteria to prioritize which hypothesis to test first — preventing confirmation bias and hypothesis fatigue.

Scoring Criteria (each scored 1–5)

Criterion 1
Explains ALL IS
5=All, 3=Most, 1=Few
Criterion 2
Explains NO IS NOT
5=Excludes all, 1=Few
Criterion 3
Change Correlation
5=Direct, 3=Indirect, 1=None
Criterion 4
Technical Plausibility
5=Proven, 3=Possible, 1=Unlikely
Criterion 5
Evidence Availability
5=Easy, 3=Moderate, 1=Hard

Hypothesis Explains IS Explains NOT Change Match Plausibility Evidence TOTAL

Network congestion 2112511

Decision: Test "Cloud EUM timeout" first (24/25). Then "HSS v3.5 bug" (21/25) if first hypothesis fails testing.

Anti-Pattern Warning

Require minimum 3 candidate hypotheses with MPC scoring before accepting any root cause. Declaring the first hypothesis as the root cause without scoring alternatives is Hypothesis Fatigue — a common reviewer rejection trigger.

Root Cause Validation

Evidence Quality Hierarchy

Root cause conclusions are only as strong as the evidence supporting them. Volume of evidence does not substitute for quality. Three log queries from the same system is still single-source evidence.

TIER 1 — STRONG

Reproducible & Definitive

Reproducible experiments, logs, config diffs, code traces. Direct proof of causal mechanism.

Mandatory for validation
TIER 2 — MODERATE

Correlational & Statistical

Time-aligned metrics, statistical correlations, controlled comparisons. Supports causality but does not prove mechanism.

Acceptable with Tier 1
TIER 3 — WEAK

Expert Opinion & Anecdotal

Expert opinion, anecdotal observations, screenshots. Directional only — useful for generating hypotheses, not validating them.

Insufficient alone
Non-Violable Evidence Standard

A validated root cause MUST be supported by: (1) at least one Tier 1 evidence source, AND (2) at least one additional independent evidence source (Tier 1 or 2). Tier 3 evidence may guide hypothesis generation but must NEVER be used to validate a root cause.

Root Cause Statement Template

ROOT CAUSE OF OCCURRENCE: The root cause of [PROBLEM] is [SPECIFIC CONDITION/CHANGE] which resulted in [TECHNICAL MECHANISM] because [UNDERLYING REASON]. VALIDATION EVIDENCE: 1. [Tier 1 evidence — e.g., log trace showing EUM timeout at profile download] 2. [Additional evidence source — e.g., lab reproduction at timeout thresholds] 3. [Additional corroboration — e.g., correlation between EUM latency and failure rate] ROOT CAUSE OF ESCAPE: The problem escaped because [DETECTION FAILURE REASON]. VALIDATION EVIDENCE: 1. [Evidence of why monitoring/testing failed to catch this]
Testing Protocol

Hypothesis Validation Steps

Once MPC is identified, design a structured test before declaring root cause. Declare the test design before running it — changing the test after seeing the data is confirmation bias.

1

Log Analysis

Examine EUM response times in activation logs during failure windows. Identify timeout events and correlate with activation failure timestamps.

2

Correlation Test

Do activation failures correlate with EUM timeout events? Use Python (pearsonr or time-series alignment) to quantify the relationship. >0.7 correlation with p < 0.05 is strong evidence.

3

Reproduction Test

Can we reproduce failures by simulating EUM delay in a controlled environment? Controlled lab reproduction is Tier 1 evidence. If reproducible, the causal mechanism is confirmed.

4

Elimination Test

Does routing around the EUM prevent failures? If bypassing the suspected cause eliminates the problem, that is near-conclusive validation. This is also your containment action prototype.

When Tests Fail

If the top MPC hypothesis fails testing, return to the MPC scoring table and test the next-ranked hypothesis. Do not retrofit evidence to a failing hypothesis. Escalate if you're stuck on all hypotheses after 2 weeks of D4 — this is the Timebox threshold (see Governance page).

Exit Criteria for D4 (Non-Violable)

Reviewer must approve ALL of the following before D5 solution design begins:

  • Root cause of OCCURRENCE identified and validated with evidence meeting quality standards
  • Root cause of ESCAPE identified and validated with evidence
  • Both root causes documented using the root cause statement template format
  • Evidence artifacts attached (logs, test results, reproduction steps, code traces)
  • Minimum 3 independent pieces of evidence for each root cause (at least one Tier 1)
  • Minimum 3 hypotheses considered with documented MPC scoring
  • IS/IS NOT table completed with all four dimensions
  • Peer review completed and approved by Reviewer/Gatekeeper
