Purpose & Positioning

What This Playbook Is

  • A decision framework for when to apply structured vs ad-hoc analysis
  • A disciplined sequence from symptom to validated root cause
  • A set of KT analytical tools integrated into 8D stages
  • A quality gate system to prevent premature closure
  • GenAI prompt templates to accelerate analysis workflows
  • Industry-tested case studies across Telecom, Retail, and Banking

What This Playbook Is NOT

  • A quick fix template or checklist substitute for thinking
  • A replacement for domain expertise and subject matter knowledge
  • For one-off issues with clear, explainable causes
  • For routine operational issues covered by standard procedures
  • A documentation exercise — every step must add analytical value
  • Optional. When triggered, governance rules are non-violable

When to Launch an 8D + KT Investigation

MANDATORY Launch Triggers

  • Customer-impacting issues >24 hours unresolved
  • P0/P1 incidents with unknown root cause
  • KPI deviation >10% from baseline for 3+ days
  • Issues with regulatory or financial implications
  • Recurring failures (>2 occurrences in 30 days)
  • Cross-functional problems with no clear owner

Optional (Judgment-Based) Triggers

  • Learning opportunities with documentation value
  • Pattern of minor issues suggesting systemic root cause
  • Stakeholders requesting formal RCA documentation
  • P2/P3 issues escalating in frequency
  • Pre-emptive analysis before known risk events

Do NOT Launch 8D When

  • One-off with clear, explainable cause
  • Pure execution gap; fix is known
  • Cost of analysis exceeds future risk
  • Routine ops handled by standard procedures
  • User error or training gap (UX improvement instead)

Non-Violable Governance Rules

RULE 1

Every 8D + KT analysis must have ONE named Problem Owner and ONE named Reviewer assigned at launch. No exceptions. The launch decision (Proceed / Redirect / Reject) must be documented BEFORE any deep analysis begins.

RULE 2

Reviewer rejections MUST be documented with specific, actionable rationale — not just "needs more work." Reviewers must state exactly what is missing and what is required to pass the gate.

RULE 3

No stage can be skipped. If a stage is deemed "not applicable," this must be explicitly documented with justification reviewed and approved by the gatekeeper.

RULE 4

No root cause, no solution. You must identify and validate BOTH the root cause of occurrence AND the root cause of escape (why detection failed) before progressing to D5.
